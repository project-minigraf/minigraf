# Design: Extract Manual Additions from release.yml

**Date:** 2026-04-26  
**Status:** Approved

## Problem

`release.yml` is auto-generated by cargo-dist. Over time, manual additions have accumulated in it:
- `docs-check` job (gates `host`)
- `publish-python` job (calls `python-release.yml` via `workflow_call`)
- `publish-c` job (calls `c-release.yml` via `workflow_call`)
- `publish-node` job (calls `node-release.yml` via `workflow_call`)

Any cargo-dist regeneration silently clobbers these, causing startup failures or broken releases (as witnessed with the v0.22.0–v0.24.0 tags having the stale `pull_request:` trigger).

## Policy

`release.yml` is cargo-dist territory. It must never be manually edited. All custom CI logic lives in standalone workflow files triggered by `push: tags`.

## Design

### New file: `docs-check.yml`

- Triggers: `push: tags` + `workflow_dispatch`
- Runs: `cargo doc --all-features --no-deps` with `RUSTDOCFLAGS: "--cfg docsrs"`
- Role: parallel check — fails visibly if docs break, but does not gate the release
- Rationale: docs failures are already caught by CI on PRs before any tag is created; the hard gate in `release.yml` provided defence-in-depth that is acceptable to lose in exchange for eliminating the fragility

### Modified: `python-release.yml`

- Add `push: tags` trigger (same pattern as `wasm-release.yml`)
- Drop `workflow_call` trigger
- Keep `workflow_dispatch` for manual re-runs
- No wait loop needed: publishes directly to PyPI, independent of the GitHub Release

### Modified: `c-release.yml`

- Add `push: tags` trigger
- Drop `workflow_call` trigger
- Keep `workflow_dispatch` for manual re-runs
- Retain existing wait-for-release loop: uploads artifacts to GitHub Releases, which must exist first

### Modified: `node-release.yml`

- Add `push: tags` trigger
- Drop `workflow_call` trigger
- Keep `workflow_dispatch` for manual re-runs
- No wait loop needed: publishes to npm, independent of the GitHub Release

### Modified: `java-release.yml`

- Add `push: tags` trigger
- Drop `workflow_call` trigger (was never called from anywhere — automation gap fixed)
- Keep `workflow_dispatch` for manual re-runs
- No wait loop needed: publishes to Maven Central, independent of the GitHub Release

### Modified: `release.yml` (one-time cleanup)

- Remove `docs-check` job
- Remove reference to `docs-check` from `host` job's `needs` list and `if` condition
- Remove `publish-python` job
- Remove `publish-c` job
- Remove `publish-node` job
- After this cleanup, `release.yml` matches what cargo-dist would generate and is never manually edited again

## Tradeoffs

| Concern | Resolution |
|---|---|
| `docs-check` no longer gates release | Acceptable: docs are checked on every PR; failures surface as a visible failing workflow on the tag |
| Java automation gap | Fixed as part of this change |
| `workflow_call` consumers removed | None existed outside `release.yml`; safe to remove |
