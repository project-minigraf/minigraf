# Phase 6.5 Design: On-Disk B+Tree Indexes

**Date**: 2026-03-23
**Status**: Approved
**Version target**: v0.9.0
**File format**: v6

---

## Goal

Replace the current paged-blob index serialisation (`btree.rs`) with proper on-disk B+tree pages so that index memory usage is O(cache_pages) instead of O(facts), open time is O(1), and checkpoint write amplification only touches modified index pages.

---

## Background

The current `btree.rs` is a paged-blob serialiser: it serialises the full in-memory `BTreeMap` to a postcard byte string, splits it into 4KB chunks, and writes them to contiguous pages. On open, the entire blob is read back and deserialised into a `BTreeMap`. At 1M facts this consumes ~1.33 GB peak heap (measured in Phase 6.4b benchmarks).

A proper on-disk B+tree maps each node to exactly one 4KB page. Lookups traverse pages on demand through the existing `PageCache`. Range scans follow the leaf-page chain. This makes index memory proportional to the number of cached pages, not the number of facts.

---

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Pending index strategy | In-memory BTreeMap for pending only; merge to on-disk B+tree at checkpoint | Matches WAL semantics; clean separation |
| Key encoding in pages | Variable-length (postcard) with slot directory | Consistent with packed fact pages; handles String attributes |
| Checkpoint write strategy | Bulk-rebuild (merge-sort) | Simpler than incremental insert; no split-bug risk; write amplification no worse than today |
| Query architecture | New `CommittedIndexReader` trait on `FactStorage` | Mirrors `CommittedFactReader`; keeps `FactStorage` backend-agnostic |
| Auto-checkpoint | Existing `wal_checkpoint_threshold` field in `OpenOptions` — no new field | Field already exists in `db.rs`; this phase ensures it triggers the new B+tree checkpoint correctly |
| Dead pages | Abandoned (not reclaimed) | Acceptable for Phase 6.5; compaction is a future concern |
| v5→v6 migration | Eager on open (not deferred to first checkpoint) | Ensures memory benefits take effect immediately without requiring a checkpoint |
| `index_checksum` in v6 | Retained with same semantics (CRC32 over committed fact pages) | Preserves corruption-detection on open; triggers full B+tree rebuild on mismatch |
| `CommittedIndexReader` end bound | `Option<&K>` — `None` means unbounded upper end | Clean open-ended scan API; bounded scans use explicit end keys |
| Backend locking in range scan | Mutex held for the full multi-page traversal | Consistent with `CommittedFactLoaderImpl::resolve`; `get_or_load` takes `&dyn StorageBackend` satisfied by the held `MutexGuard`. Cache hits don't call `backend.read_page`, so hold time is short for warm caches. **Revisit** if benchmarks show read-read contention — per-page lock acquire/release would allow true concurrent scans. |
| `fact_page_count` in header | New field in v6 header (80 bytes, up from 72) | Replaces fragile `min(root_pages) - 1` heuristic; required for correct CRC computation after dead pages accumulate |

| Component | Before | After |
|-----------|--------|-------|
| `FactStorage.indexes` | All facts (pending + committed) | Pending facts only (renamed to `pending_indexes`) |

## B+Tree Page Format

### Leaf Node Page (type `0x21`)

```
Offset  Size                    Field
0       1                       page_type = 0x21
1       1                       reserved
2       2                       entry_count (u16)
4       8                       next_leaf (u64 page_id; 0 = last leaf)
12      entry_count × 4 bytes   slot directory: [(offset u16, length u16)]
...     variable                postcard-serialised (Key, FactRef) pairs, written end-to-start
```

### Internal Node Page (type `0x22`)

A node with `key_count` separator keys has exactly `key_count + 1` child pointers:
- `rightmost_child` at offset 4: the child page for keys ≥ the last separator
- Child array at offset 12: `key_count` left-child page IDs — `child[i]` is the subtree for keys < `separator[i]`

```
Offset  Size                       Field
0       1                          page_type = 0x22
1       1                          reserved
2       2                          key_count (u16)
4       8                          rightmost_child (u64 page_id) — subtree for keys > all separators
12      key_count × 8 bytes        child page IDs: child[i] is subtree for keys < separator[i]
12 + key_count×8                   key_count × 4 bytes — slot directory: [(offset u16, length u16)]
...     variable                   postcard-serialised separator Key bytes, written end-to-start
```

Total child pointers = `key_count` (in child array) + 1 (`rightmost_child`) = `key_count + 1`. ✓

**Traversal rule**: to descend from an internal node for a search key `K`:
- For `i` in `0..key_count`: if `K < separator[i]` → descend into `child[i]`
- If no separator matched → descend into `rightmost_child`

**Fill factor — leaves and internal nodes**: during bulk-build, stop adding entries to a page once the total data bytes written to that page exceed 3KB (~75% of 4KB). This rule applies identically to both leaf pages and internal node pages. Each internal node entry costs 8 bytes (child pointer) + 4 bytes (slot directory entry) + serialised key bytes; stop when the running total exceeds 3KB.

**Separator key convention**: `separator[i]` is the first key of `leaf[i+1]` (copy-up style). The first leaf has no separator above it.

## New Module: `src/storage/btree_v6.rs`

Four logical sections:

### 1. Builder (`build_btree`)

```rust
pub fn build_btree<K>(
    sorted_entries: impl Iterator<Item = (K, FactRef)>,
    backend: &mut dyn StorageBackend,
    cache: &PageCache,
    start_page_id: u64,
) -> Result<(u64, u64)>  // (root_page_id, next_free_page_id)
where
    K: Serialize + Ord,
```

Returns `(root_page_id, next_free_page_id)` so callers can chain multiple `build_btree` calls (one per index) by passing the returned `next_free_page_id` as `start_page_id` for the next call.

All pages written by `build_btree` are passed to `cache.put_dirty(page_id, data)` immediately after writing to the backend, so subsequent `range_scan` calls find them in the cache without a backend re-read.

Algorithm:
1. Pack entries into leaf pages (fill factor 75% per page), linking each leaf to the next via `next_leaf`. Call `backend.write_page` + `cache.put_dirty` for each page. Assign page IDs sequentially starting at `start_page_id`.
2. Collect separator keys: for each leaf except the first, record its first key as the separator between it and the previous leaf.
3. Build internal node pages bottom-up from separator keys + child page IDs (fill factor 75% per page). Call `backend.write_page` + `cache.put_dirty` for each page.
4. If only one leaf was written and no internal nodes were needed, return `(leaf_page_id, leaf_page_id + 1)`.
5. Otherwise return `(root_page_id, last_page_id + 1)` where `root_page_id` is the topmost internal node.

### 2. Leaf Streamer (`stream_all_entries`)

```rust
pub fn stream_all_entries<K>(
    root_page_id: u64,
    backend: &dyn StorageBackend,
    cache: &PageCache,
) -> Result<Vec<(K, FactRef)>>
where
    K: for<'de> Deserialize<'de> + Ord,
```

Traverses internal nodes from the root to find the first (leftmost) leaf, then follows the `next_leaf` chain to collect all `(K, FactRef)` pairs in sorted order. All page reads go through `cache.get_or_load(page_id, backend)`. Used during checkpoint to stream committed index entries for the merge-sort step.

### 3. Range Scanner (`range_scan`)

```rust
pub fn range_scan<K>(
    root_page_id: u64,
    start: &K,
    end: Option<&K>,
    backend: &dyn StorageBackend,
    cache: &PageCache,
) -> Result<Vec<FactRef>>
where
    K: Serialize + for<'de> Deserialize<'de> + Ord,
```

`end: None` means unbounded upper end (scan to the last leaf). All page reads go through `cache.get_or_load(page_id, backend)`. No direct `backend.read_page` calls.

Algorithm:
1. Traverse internal nodes from root to find the leaf containing `start` (applying traversal rule).
2. Scan leaf pages following `next_leaf` chain, collecting `FactRef`s where `start <= key` and (`end.is_none()` or `key <= end`).
3. Stop when `key > end` (if bounded) or `next_leaf == 0`.

### 4. `OnDiskIndexReader`

Implements `CommittedIndexReader`. Created inside `PersistentFactStorage<B>` (fixing `B`), then boxed as `Box<dyn CommittedIndexReader>` — same type-erasure pattern as `CommittedFactLoaderImpl<B>`.

Fields:
- `Arc<PageCache>`
- `Arc<Mutex<B>>` where `B: StorageBackend + 'static`
- `eavt_root`, `aevt_root`, `avet_root`, `vaet_root: u64`

Each `range_scan_*` method: lock the backend mutex for the full traversal, then call `range_scan(root, start, end, &*guard, cache)`. The lock is held for the entire multi-page traversal (consistent with `CommittedFactLoaderImpl::resolve`; `get_or_load` accepts `&dyn StorageBackend` satisfied by the held `MutexGuard`).

## New Trait: `CommittedIndexReader`

Added to `src/storage/mod.rs` alongside `CommittedFactReader`:

```rust
pub trait CommittedIndexReader: Send + Sync {
    fn range_scan_eavt(&self, start: &EavtKey, end: Option<&EavtKey>) -> Result<Vec<FactRef>>;
    fn range_scan_aevt(&self, start: &AevtKey, end: Option<&AevtKey>) -> Result<Vec<FactRef>>;
    fn range_scan_avet(&self, start: &AvetKey, end: Option<&AvetKey>) -> Result<Vec<FactRef>>;
    fn range_scan_vaet(&self, start: &VaetKey, end: Option<&VaetKey>) -> Result<Vec<FactRef>>;

```

`end: None` means unbounded upper end. No sentinel values are needed.

## `FactStorage` Changes

### Field Changes

```rust
// Before
indexes: Indexes,                                              // all facts
committed_fact_reader: Option<Arc<dyn CommittedFactReader>>,

// After
pending_indexes: Indexes,                                      // pending only (renamed)
committed_fact_reader: Option<Arc<dyn CommittedFactReader>>,
committed_index_reader: Option<Arc<dyn CommittedIndexReader>>, // new
```

**New setter method**: `FactStorage::set_committed_index_reader(reader: Arc<dyn CommittedIndexReader>)` — sets `committed_index_reader`. Called by `persistent_facts.rs` after building `OnDiskIndexReader` during open, migration, checkpoint, and rebuild.

**All call sites referencing `indexes` / `FactData.indexes` must be updated to `pending_indexes`.** Affected items:
- `FactStorage::insert` / `load_fact` — insert into `pending_indexes`
- `FactStorage::replace_indexes` → renamed `replace_pending_indexes`
- `FactStorage::indexes_snapshot` → renamed `pending_indexes_snapshot`
- `persistent_facts.rs` — remove any code that loads committed `FactRef`s into `indexes` after open (this is replaced by calling `set_committed_index_reader`)

### Post-Checkpoint Clearing

`clear_pending_facts()` on `FactStorage` currently clears `d.facts` only. It must be extended (or replaced by a `post_checkpoint_clear()` method) to also clear `d.pending_indexes`. Checkpoint step 8 calls this combined clear.

### Query Merge Pattern (EAVT example)

All committed index range scans must use a **bounded end key** to avoid scanning the entire index. For EAVT entity scans, the end key is constructed using the next UUID after the target entity as the exclusive upper bound — all keys with `key.entity == target_entity` sort strictly before `(next_entity, "", i64::MIN, i64::MIN, 0)`:

```rust
// Get all facts for a given entity — EAVT scan
let start = EavtKey { entity, attribute: String::new(), valid_from: i64::MIN,
                      valid_to: i64::MIN, tx_count: 0 };

// Exclusive upper bound: first key of the next entity in UUID order
let next_entity = Uuid::from_u128(entity.as_u128().wrapping_add(1));
let end = EavtKey { entity: next_entity, attribute: String::new(),
                    valid_from: i64::MIN, valid_to: i64::MIN, tx_count: 0 };

// Pending: in-memory BTreeMap, bounded range
let pending_refs: Vec<FactRef> = pending_indexes.eavt
    .range(start..end)    // exclusive end: all entity keys, none of next_entity
    .map(|(_, r)| *r)
    .collect();

// Committed: on-disk B+tree, bounded end
let committed_refs: Vec<FactRef> = committed_index_reader
    .range_scan_eavt(&start, Some(&end))?;

// Resolve: page_id == 0 → pending vec; page_id >= 1 → CommittedFactReader
```

**Upper-bound construction per query type:**
- **EAVT entity scan**: end = `EavtKey { entity: Uuid::from_u128(entity.as_u128().wrapping_add(1)), attribute: String::new(), valid_from: i64::MIN, valid_to: i64::MIN, tx_count: 0 }`
- **AEVT attribute scan**: end = `AevtKey { attribute: next_string_prefix(attr), entity: Uuid::nil(), valid_from: i64::MIN, valid_to: i64::MIN, tx_count: 0 }` where `next_string_prefix` increments the last byte
- **AVET attribute+value scan**: bounded by attribute and value bytes — same prefix-increment pattern
- **VAET ref-target scan**: end = `VaetKey { ref_target: Uuid::from_u128(ref_target.as_u128().wrapping_add(1)), ... }`

`end: None` is reserved for truly unbounded scans (e.g. `stream_all_entries` is preferred for that case).

**Full scans** (no index hint): fall back to `CommittedFactReader.stream_all()` + pending vec. No index needed.

**`load_fact` (WAL replay)**: inserts into `pending_indexes` only. Unchanged.

## Checkpoint Sequence

After writing pending facts to fact pages (step 1), `backend.page_count()` gives `next_free_page_id` — the first available page ID for index data. The four `build_btree` calls are chained using the returned `next_free_page_id`:

1. Pack pending facts → new fact pages (unchanged from v5)
2. `next_free = backend.page_count()` after step 1
3. For each index in order (EAVT, AEVT, AVET, VAET):
   - If existing B+tree root ≠ 0: call `stream_all_entries(root, backend, cache)` → sorted committed entries
   - Else: committed entries = empty
   - Merge-sort committed entries with sorted pending BTreeMap entries for this index
   - `(root, next_free) = build_btree(merged_iter, backend, cache, next_free)`
   - Record new root page ID
4. Compute new `index_checksum` (CRC32 over committed fact pages — same as v5 logic)
5. Write `FileHeader` v6: version=6, new root page IDs, `page_count = next_free`, `fact_page_count = num_new_fact_pages`, updated `index_checksum`
6. Sync backend
7. Clear WAL
8. Replace `committed_index_reader` on `FactStorage` with new `OnDiskIndexReader` (new root page IDs)
9. Call `post_checkpoint_clear()` on `FactStorage` (clears `pending_facts` + `pending_indexes`)

Old index pages are abandoned after step 5. They are no longer referenced by the header and will never be loaded into the page cache.

## Checksum-Mismatch Rebuild Path (v6)

The existing v5 `reindex_with_refs` path populates `indexes` with committed `FactRef`s from memory — this is removed in v6. The v6 rebuild path:

1. `stream_all()` via `CommittedFactReader` → all committed `Fact` objects
2. Build 4 sorted iterators of `(Key, FactRef)` from those facts (same logic as `reindex_from_facts`)
3. `next_free = 1 + header.fact_page_count` (page immediately after the last committed fact page)
4. Call `build_btree` for each index (chained `next_free_page_id`) → new B+tree pages
5. Write updated `FileHeader`: new root page IDs, `page_count = final_next_free`, `fact_page_count` unchanged, new `index_checksum`
6. Sync backend
7. Set `committed_index_reader` to new `OnDiskIndexReader`

This is the v6 equivalent of `reindex_with_refs`. The old function is removed; v6 uses this new path whenever `index_checksum` mismatches on open.

## Auto-Checkpoint

Auto-checkpoint is already implemented via `OpenOptions::wal_checkpoint_threshold` (`db.rs:58`, default 1,000 facts). After `WriteTransaction::commit()`, if the WAL entry count ≥ threshold, `checkpoint()` is called synchronously on the committing thread.

**No new field or API is added.** This phase verifies that the existing auto-checkpoint path correctly triggers the new B+tree checkpoint (i.e., `self.checkpoint()` in `db.rs` calls the updated `PersistentFactStorage::save()` that performs the B+tree merge). No changes to `OpenOptions` or `commit()` logic are required beyond what the rest of Phase 6.5 already touches.

## v5 → v6 Migration

Migration is **eager on open** (not deferred to first checkpoint), so memory benefits take effect immediately. New B+tree pages are written starting at the current `header.page_count`, beyond all existing content. If the process crashes mid-migration, the header has not yet been updated to v6, so the file remains a valid v5 file on re-open and migration restarts cleanly from step 1. This is safe and idempotent.

**On open of a v5 file:**
1. Read old paged-blob indexes via `btree.rs` `read_*_index` functions → 4 in-memory BTreeMaps
2. `next_free = header.page_count`
3. For each index (EAVT, AEVT, AVET, VAET):
   - `(root, next_free) = build_btree(sorted_iter_of_btreemap, backend, cache, next_free)`
   - Record new root page ID
4. Write v6 header (version=6) with new root page IDs, `page_count = next_free`, `fact_page_count = min(old_root_pages) - 1` (derived once from v5 heuristic during migration only); `index_checksum` unchanged
5. Sync backend
6. Set `committed_index_reader` to new `OnDiskIndexReader`
7. `pending_indexes` starts empty

**`btree.rs`** is kept read-only for this migration path. It can be removed in a future phase.

## `FileHeader` v6

**Expanded to 80 bytes** (was 72). Layout:

```
Offset  Size  Field
0..4    4     magic ("MGRF")
4..8    4     version (u32) = 6
8..16   8     page_count (u64)
16..24  8     node_count / fact count (u64)
24..32  8     last_checkpointed_tx_count (u64)
32..40  8     eavt_root_page (u64)
40..48  8     aevt_root_page (u64)
48..56  8     avet_root_page (u64)
56..64  8     vaet_root_page (u64)
64..68  4     index_checksum (u32)
68      1     fact_page_format (u8)
69..72  3     _padding ([u8; 3])
72..80  8     fact_page_count (u64)   ← NEW in v6
```

`fact_page_count` records exactly how many pages (starting at page 1) hold committed fact data. Replaces the fragile `min(root_pages) - 1` heuristic that breaks after dead pages accumulate.

Changes from v5:
- `FORMAT_VERSION = 6`
- `FileHeader` struct gains `fact_page_count: u64`
- `to_bytes()` emits 80 bytes; `from_bytes` reads `fact_page_count` from bytes 72..80 for version 6 headers
- v1–v5 `from_bytes` paths unchanged; `fact_page_count` is zero-initialised for older versions (derived from heuristic on migration)
- `validate()` accepts versions 1–6
- Tests updated: `test_validate_rejects_version_0_and_6` → `test_validate_rejects_version_0_and_7`; `test_format_version_is_5` → `test_format_version_is_6`; `test_new_header_has_version_5` → `test_new_header_has_version_6`
- New byte-layout pin test for v6 (80-byte layout)

`index_checksum` retained: CRC32 over committed fact pages. Mismatch triggers the v6 rebuild path.

## Testing Strategy

### Unit Tests

**`src/storage/btree_v6.rs`**:
- `build_btree` with 0, 1, and N entries — correct `(root, next_free)` returned; leaf chain intact; all written pages appear in cache after build
- `stream_all_entries` — returns all entries in sorted order; round-trips with `build_btree`
- Range scan: exact match, prefix range, empty range (`start > all keys`), unbounded end (`end = None`), range spanning multiple leaf pages
- Fill-factor: leaf pages and internal node pages do not exceed 3KB of data
- Multi-level tree: enough entries to force internal node creation; verify traversal uses only `cache.get_or_load`
- Traversal rule: verify `K < separator[i]` → `child[i]`, else `rightmost_child`
- Chained `build_btree` calls: verify `next_free_page_id` from call N equals `start_page_id` for call N+1 and no page IDs overlap
- `OnDiskIndexReader`: all four `range_scan_*` methods with `MemoryBackend`; unbounded end scan

**`src/storage/mod.rs`**:
- `FileHeader` v6 byte-layout pin test
- `validate()` accepts 1–6, rejects 0 and 7
- `test_validate_rejects_version_0_and_7` (renamed from `test_validate_rejects_version_0_and_6`)

### Integration Tests (`tests/btree_v6_test.rs`)

- **Roundtrip**: transact facts → checkpoint → close → reopen → query returns correct results
- **Range scan across leaves**: 500+ facts; EAVT range returns correct subset
- **Unbounded scan**: query with no upper bound returns all matching facts
- **v5→v6 migration**: create v5 file → open with v6 code → migration runs eagerly → queries correct → header version=6
- **Migration crash-safety**: simulate crash mid-migration (write B+tree pages but not header) → re-open still sees v5 → migration runs again → correct result
- **Pending + committed merge**: transact → checkpoint (committed) → transact more (pending) → query sees both sets
- **Auto-checkpoint**: transact facts exceeding threshold → WAL cleared automatically after commit
- **Dead space**: after two checkpoints, old B+tree root page IDs not referenced by header; `page_count` advances monotonically; queries correct
- **Checksum-mismatch rebuild (v6)**: corrupt `index_checksum` in header of a v6 file → open triggers v6 rebuild path → queries return correct results; header has updated checksum

### Concurrency Benchmark (`benches/`)

Add a dedicated benchmark: N threads (2, 4, 8) each running simultaneous EAVT range scans against a pre-populated committed B+tree (facts already checkpointed, page cache partially warm). Measure total throughput (scans/sec) as N increases. A near-linear drop in per-thread throughput as N grows signals backend mutex contention and would trigger the per-page locking revisit noted in the design decisions.

### Regression

All 301 existing tests must continue to pass (with the one renamed test noted above).
