# Phase 6.5 Design: On-Disk B+Tree Indexes

**Date**: 2026-03-23
**Status**: Approved
**Version target**: v0.9.0
**File format**: v6

---

## Goal

Replace the current paged-blob index serialisation (`btree.rs`) with proper on-disk B+tree pages so that index memory usage is O(cache_pages) instead of O(facts), open time is O(1), and checkpoint write amplification only touches modified index pages.

---

## Background

The current `btree.rs` is a paged-blob serialiser: it serialises the full in-memory `BTreeMap` to a postcard byte string, splits it into 4KB chunks, and writes them to contiguous pages. On open, the entire blob is read back and deserialised into a `BTreeMap`. At 1M facts this consumes ~1.33 GB peak heap (measured in Phase 6.4b benchmarks).

A proper on-disk B+tree maps each node to exactly one 4KB page. Lookups traverse pages on demand through the existing `PageCache`. Range scans follow the leaf-page chain. This makes index memory proportional to the number of cached pages, not the number of facts.

---

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Pending index strategy | In-memory BTreeMap for pending only; merge to on-disk B+tree at checkpoint | Matches WAL semantics; clean separation |
| Key encoding in pages | Variable-length (postcard) with slot directory | Consistent with packed fact pages; handles String attributes |
| Checkpoint write strategy | Bulk-rebuild (merge-sort) | Simpler than incremental insert; no split-bug risk; write amplification no worse than today |
| Query architecture | New `CommittedIndexReader` trait on `FactStorage` | Mirrors `CommittedFactReader`; keeps `FactStorage` backend-agnostic |
| Auto-checkpoint | WAL size threshold, synchronous, configurable | Prevents unbounded WAL growth; no background thread needed |
| Dead pages | Abandoned (not reclaimed) | Acceptable for Phase 6.5; compaction is a future concern |
| v5→v6 migration | Eager on open (not deferred to first checkpoint) | Ensures memory benefits take effect immediately without requiring a checkpoint |
| `index_checksum` in v6 | Retained with same semantics (CRC32 over committed fact pages) | Preserves corruption-detection on open; triggers full index rebuild on mismatch |

## Architecture Overview

### What Changes

| Component | Before | After |
|-----------|--------|-------|
| `FactStorage.indexes` | All facts (pending + committed) | Pending facts only (renamed to `pending_indexes`) |
| `FactStorage` | Takes `CommittedFactReader` | Also takes `CommittedIndexReader` |
| `btree.rs` | Paged-blob serialiser | Kept read-only for v5 migration; no new writes |
| `btree_v6.rs` | (new) | B+tree builder, range scanner, `OnDiskIndexReader` |
| `persistent_facts.rs` | Loads full index into RAM on open | Sets `CommittedIndexReader` on `FactStorage`; no index load |
| Checkpoint | Serialise BTreeMap → pages | Merge pending BTreeMap + existing B+tree → new B+tree pages |
| `FileHeader` | v5 | v6 (same 72-byte layout, version byte = 6) |
| Index page types | `0x11` (paged blob) | `0x21` (leaf node), `0x22` (internal node) |
| `FORMAT_VERSION` constant | `5` | `6` |
| `test_validate_rejects_version_0_and_6` | Asserts version 6 is rejected | Must be renamed/updated: version 7 is rejected; version 6 is accepted |

### What Is Unchanged

- `StorageBackend` trait
- `PageCache`
- `FactRef` struct
- Key types (`EavtKey`, `AevtKey`, `AvetKey`, `VaetKey`)
- `CommittedFactReader` trait
- WAL format and crash-recovery logic
- `WriteTransaction` public API (commit() gains auto-checkpoint check internally)
- `index_checksum` semantics (CRC32 over committed fact pages; mismatch triggers rebuild)

## B+Tree Page Format

### Leaf Node Page (type `0x21`)

```
Offset  Size                    Field
0       1                       page_type = 0x21
1       1                       reserved
2       2                       entry_count (u16)
4       8                       next_leaf (u64 page_id; 0 = last leaf)
12      entry_count × 4 bytes   slot directory: [(offset u16, length u16)]
...     variable                postcard-serialised (Key, FactRef) pairs, written end-to-start
```

### Internal Node Page (type `0x22`)

A node with `key_count` separator keys has exactly `key_count + 1` child pointers:
- `rightmost_child` at offset 4: the child page for keys greater than all separator keys
- Child array at offset 12: `key_count` left-child page IDs — child[i] is the subtree for keys less than separator[i]

```
Offset  Size                    Field
0       1                       page_type = 0x22
1       1                       reserved
2       2                       key_count (u16)
4       8                       rightmost_child (u64 page_id) — child for keys > all separators
12      key_count × 8 bytes     child page IDs: child[i] is subtree for keys < separator[i]
12 + key_count×8                key_count × 4 bytes — slot directory: [(offset u16, length u16)]
...     variable                postcard-serialised separator Key bytes, written end-to-start
```

Total child pointers = `key_count` (in child array) + 1 (`rightmost_child`) = `key_count + 1`. ✓

**Traversal rule**: to descend from an internal node for a search key `K`:
- For i in 0..key_count: if `K < separator[i]` → descend into `child[i]`
- If no separator matched → descend into `rightmost_child`

**Fill factor**: ~75% during bulk-build (stop adding entries once page data exceeds 3KB). Leaves ~1KB headroom for a future incremental-insert phase.

**Separator key convention**: separator[i] is the first key of leaf[i+1] (copy-up style). The first leaf has no separator above it.

## New Module: `src/storage/btree_v6.rs`

Three logical sections:

### 1. Builder (`build_btree`)

```rust
pub fn build_btree<K>(
    sorted_entries: impl Iterator<Item = (K, FactRef)>,
    backend: &mut dyn StorageBackend,
    cache: &PageCache,
    start_page_id: u64,
) -> Result<u64>  // returns root page ID
where K: Serialize + Ord
```

All pages written by `build_btree` are passed to `cache.put_dirty(page_id, data)` immediately after writing to the backend, so subsequent `range_scan` calls find them in the cache without a backend re-read.

Algorithm:
1. Pack entries into leaf pages (fill factor 75%, linked via `next_leaf`), calling `put_dirty` for each page written
2. Collect separator keys: for each leaf except the first, record its first key as the separator between it and the previous leaf
3. Build internal node pages bottom-up from separator keys + child page IDs, calling `put_dirty` for each
4. If only one leaf, return that leaf page ID as root (no internal nodes needed)
5. Return root page ID

### 2. Range Scanner (`range_scan`)

```rust
pub fn range_scan<K>(
    root_page_id: u64,
    start: &K,
    end: &K,
    backend: &dyn StorageBackend,
    cache: &PageCache,
) -> Result<Vec<FactRef>>
where K: Serialize + Deserialize + Ord
```

**All page reads go through `cache.get_or_load(page_id, backend)`.** Direct `backend.read_page` calls are not used.

Algorithm:
1. Traverse internal nodes from root: at each node apply the traversal rule (`K < separator[i]` → child[i], else `rightmost_child`) until a leaf node is reached
2. Scan leaf pages following `next_leaf` chain, collecting `FactRef`s for entries where `start <= key <= end`
3. Stop when `key > end` or `next_leaf == 0`

### 3. `OnDiskIndexReader`

Implements the new `CommittedIndexReader` trait. Created inside `PersistentFactStorage<B>` (which fixes the backend type `B`), then stored as `Box<dyn CommittedIndexReader>` — same type-erasure pattern used by `CommittedFactLoaderImpl<B>` for `CommittedFactReader`.

Fields:
- `Arc<PageCache>`
- `Arc<Mutex<B>>` where `B: StorageBackend + 'static` (same `Arc<Mutex<B>>` shared with `CommittedFactLoaderImpl`)
- `eavt_root`, `aevt_root`, `avet_root`, `vaet_root: u64`

Each `range_scan_*` method locks the backend mutex, then calls the generic `range_scan` with the appropriate root page ID and key bounds, passing `cache` and `&*backend`.

## `FactStorage` Changes

### Field Changes

The following fields and methods on `FactStorage` (and its inner `FactData`) are affected:

```rust
// Before
indexes: Indexes,                                              // all facts
committed_fact_reader: Option<Arc<dyn CommittedFactReader>>,

// After
pending_indexes: Indexes,                                      // pending only (renamed)
committed_fact_reader: Option<Arc<dyn CommittedFactReader>>,
committed_index_reader: Option<Arc<dyn CommittedIndexReader>>, // new
```

**All call sites that reference `indexes` or `FactData.indexes` must be updated to `pending_indexes`.** This includes:
- `FactStorage::insert` / `load_fact` — inserts into `pending_indexes`
- `FactStorage::replace_indexes` / `indexes_snapshot` (if present) — renamed to `replace_pending_indexes` / `pending_indexes_snapshot`
- `persistent_facts.rs` — any call that passes committed FactRefs into `indexes` after load (removed; committed entries are no longer loaded into memory)

### Query Merge Pattern (EAVT example)

```
start = EavtKey { entity, attribute: "", valid_from: i64::MIN, valid_to: i64::MIN, tx_count: 0 }
end   = EavtKey { entity, attribute: "\u{10FFFF}", valid_from: i64::MAX, valid_to: i64::MAX, tx_count: u64::MAX }

pending_refs   = pending_indexes.eavt.range(start..=end)               // in-memory BTreeMap
committed_refs = committed_index_reader.range_scan_eavt(start, end)    // page cache

all_refs = chain(pending_refs, committed_refs)
// resolve each FactRef → Fact:
//   page_id == 0 → pending vec
//   page_id >= 1 → CommittedFactReader (page cache)
```

Same merge pattern applies to `get_facts_by_attribute` (AEVT), value-range scans (AVET), and reverse-ref scans (VAET).

**Full scans** (no index hint): fall back to `CommittedFactReader.stream_all()` + pending vec. No index needed.

**`load_fact` (WAL replay)**: inserts into `pending_indexes` only. Unchanged.

## Checkpoint Sequence

1. Pack pending facts → new fact pages (unchanged)
2. For each of the 4 indexes:
   - If existing B+tree root ≠ 0: stream its leaf pages sequentially (via `cache.get_or_load`) to get sorted committed entries
   - Merge-sort with sorted pending BTreeMap entries
   - Call `build_btree(merged_iter, backend, cache, next_free_page_id)` → new B+tree pages; get new root page ID
3. Compute new `index_checksum` (CRC32 over committed fact pages — same as v5 logic)
4. Write `FileHeader` v6 with new root page IDs, updated `page_count`, updated `index_checksum`
5. Sync backend
6. Clear WAL
7. Replace `committed_index_reader` on `FactStorage` with new `OnDiskIndexReader` (new root page IDs)
8. Clear `pending_indexes`

Old index pages (paged-blob `0x11` or previous B+tree pages) are abandoned after step 4. Their page IDs are no longer referenced by the header and will never be loaded into the page cache.

---

## Auto-Checkpoint

**Trigger**: after `WriteTransaction::commit()`, if the WAL fact count exceeds `auto_checkpoint_threshold`, synchronously call `checkpoint()`.

**Configuration**: `OpenOptions::auto_checkpoint_threshold(usize)` (default: 1,000 facts; 0 = disabled).

**Implementation**: `Minigraf` stores the threshold. `commit()` checks `wal.fact_count() >= threshold` and calls `self.checkpoint()` if true. Runs on the committing thread; no background thread.

## v5 → v6 Migration

Migration is **eager on open** (not deferred to first checkpoint), so memory benefits take effect immediately. New B+tree pages are written starting at the current `header.page_count` so they are beyond existing content; if the process crashes mid-migration, the file remains a valid v5 file (old paged-blob root page IDs in the header are still correct) and migration restarts cleanly on re-open.

**On open of a v5 file:**
1. Read old paged-blob indexes via existing `btree.rs` `read_*_index` functions → 4 in-memory BTreeMaps
2. Immediately call `build_btree` for each → write v6 B+tree pages starting at `header.page_count`; populate page cache via `put_dirty`
3. Write v6 header (version = 6) with new root page IDs; `index_checksum` unchanged (fact pages not modified)
4. Sync backend
5. Set `committed_index_reader` to new `OnDiskIndexReader`
6. `pending_indexes` starts empty

**Crash safety**: steps 2–3 write pages beyond the old `page_count`, so a partial write leaves the old header and old paged-blob pages intact. On re-open the file is still v5 (header not yet updated to v6) and migration restarts from step 1. This is safe and idempotent.

**`btree.rs`** is kept read-only so this migration path continues to work. It can be removed in a future phase once v5 files are no longer in use.

## `FileHeader` v6

Same 72-byte layout as v5. Changes:

```
FORMAT_VERSION = 6   (was 5)
```

`eavt_root_page`, `aevt_root_page`, `avet_root_page`, `vaet_root_page` continue to hold root page IDs — they pointed to paged-blob roots in v5, and now point to B+tree roots in v6.

`index_checksum` is retained with the same semantics: CRC32 over committed fact pages. On open, if `index_checksum` does not match, a full index rebuild is triggered (same as v5 behavior).

**`validate()`** updated to accept versions 1–6.

**Existing test `test_validate_rejects_version_0_and_6`** (in `src/storage/mod.rs`) must be renamed to `test_validate_rejects_version_0_and_7` and updated to assert version 7 is rejected and version 6 is accepted.

## Testing Strategy

### Unit Tests

**`src/storage/btree_v6.rs`**:
- `build_btree` with 0, 1, and N entries — root page ID returned; leaf chain intact; pages appear in cache after build
- Range scan: exact match, prefix range, empty range, range spanning multiple leaf pages
- Fill-factor: pages do not exceed 3KB of data
- Multi-level tree: enough entries to force internal node creation (>~50 entries); verify traversal via `cache.get_or_load` only
- Traversal rule correctness: verify `K < separator[i]` → child[i], else `rightmost_child`
- `OnDiskIndexReader`: all four `range_scan_*` methods with `MemoryBackend`

**`src/storage/mod.rs`**:
- `FileHeader` v6 byte-layout pin test (same pattern as v5)
- `validate()` accepts versions 1–6, rejects 0 and 7
- Rename `test_validate_rejects_version_0_and_6` → `test_validate_rejects_version_0_and_7`

### Integration Tests (`tests/btree_v6_test.rs`)

- **Roundtrip**: transact facts → checkpoint → close → reopen → query returns correct results
- **Range scan across leaves**: 500+ facts; EAVT range returns correct subset
- **v5→v6 migration**: create v5 file → open with v6 code → migration runs eagerly → queries work → header has version 6; verify crash-safety by simulating partial migration (old paged-blob pages still present, header still v5)
- **Pending + committed merge**: transact → checkpoint (committed) → transact more (pending) → query sees both
- **Auto-checkpoint**: transact facts exceeding threshold → WAL is cleared automatically after commit
- **Dead space**: after two checkpoints, verify old B+tree root page IDs are no longer referenced by the header, `page_count` advances monotonically, and queries return correct results

### Regression

All 301 existing tests must continue to pass (with the one renamed test noted above).
