# Phase 6.5 Design: On-Disk B+Tree Indexes

**Date**: 2026-03-23
**Status**: Approved
**Version target**: v0.9.0
**File format**: v6

---

## Goal

Replace the current paged-blob index serialisation (`btree.rs`) with proper on-disk B+tree pages so that index memory usage is O(cache_pages) instead of O(facts), open time is O(1), and checkpoint write amplification only touches modified index pages.

---

## Background

The current `btree.rs` is a paged-blob serialiser: it serialises the full in-memory `BTreeMap` to a postcard byte string, splits it into 4KB chunks, and writes them to contiguous pages. On open, the entire blob is read back and deserialised into a `BTreeMap`. At 1M facts this consumes ~1.33 GB peak heap (measured in Phase 6.4b benchmarks).

A proper on-disk B+tree maps each node to exactly one 4KB page. Lookups traverse pages on demand through the existing `PageCache`. Range scans follow the leaf-page chain. This makes index memory proportional to the number of cached pages, not the number of facts.

---

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Pending index strategy | In-memory BTreeMap for pending only; merge to on-disk B+tree at checkpoint | Matches WAL semantics; clean separation |
| Key encoding in pages | Variable-length (postcard) with slot directory | Consistent with packed fact pages; handles String attributes |
| Checkpoint write strategy | Bulk-rebuild (merge-sort) | Simpler than incremental insert; no split-bug risk; write amplification no worse than today |
| Query architecture | New `CommittedIndexReader` trait on `FactStorage` | Mirrors `CommittedFactReader`; keeps `FactStorage` backend-agnostic |
| Auto-checkpoint | WAL size threshold, synchronous, configurable | Prevents unbounded WAL growth; no background thread needed |
| Dead pages | Abandoned (not reclaimed) | Acceptable for Phase 6.5; compaction is a future concern |
| v5→v6 migration | Eager on open (not deferred to first checkpoint) | Ensures memory benefits take effect immediately without requiring a checkpoint |

---

## Architecture Overview

### What Changes

| Component | Before | After |
|-----------|--------|-------|
| `FactStorage.indexes` | All facts (pending + committed) | Pending facts only (`pending_indexes`) |
| `FactStorage` | Takes `CommittedFactReader` | Also takes `CommittedIndexReader` |
| `btree.rs` | Paged-blob serialiser | Kept read-only for v5 migration; no new writes |
| `btree_v6.rs` | (new) | B+tree builder, range scanner, `OnDiskIndexReader` |
| `persistent_facts.rs` | Loads full index into RAM on open | Sets `CommittedIndexReader` on `FactStorage`; no index load |
| Checkpoint | Serialise BTreeMap → pages | Merge pending BTreeMap + existing B+tree → new B+tree pages |
| `FileHeader` | v5 | v6 (same 72-byte layout, version byte = 6) |
| Index page types | `0x11` (paged blob) | `0x21` (leaf node), `0x22` (internal node) |

### What Is Unchanged

- `StorageBackend` trait
- `PageCache`
- `FactRef` struct
- Key types (`EavtKey`, `AevtKey`, `AvetKey`, `VaetKey`)
- `CommittedFactReader` trait
- WAL format and crash-recovery logic
- `WriteTransaction` API (public interface unchanged; `commit()` gains auto-checkpoint check)

## B+Tree Page Format

### Leaf Node Page (type `0x21`)

```
Offset  Size                    Field
0       1                       page_type = 0x21
1       1                       reserved
2       2                       entry_count (u16)
4       8                       next_leaf (u64 page_id; 0 = last leaf)
12      entry_count × 4 bytes   slot directory: [(offset u16, length u16)]
...     variable                postcard-serialised (Key, FactRef) pairs, written end-to-start
```

### Internal Node Page (type `0x22`)

```
Offset  Size                       Field
0       1                          page_type = 0x22
1       1                          reserved
2       2                          key_count (u16)
4       8                          rightmost_child (u64 page_id)
12      (key_count + 1) × 8 bytes  child page IDs (left child of key[0], ..., rightmost_child)
12 + (key_count+1)×8               key_count × 4 bytes — slot directory: [(offset u16, length u16)]
...     variable                   postcard-serialised Key bytes, written end-to-start
```

**Fill factor**: ~75% during bulk-build (stop adding entries once page data exceeds 3KB). Leaves ~1KB headroom for a future incremental-insert phase.

**Key comparison**: keys are deserialised from postcard bytes and compared using their derived `Ord` implementation. This is correct because postcard is deterministic and the key types derive `Ord` over their fields in sort-key order.

## New Module: `src/storage/btree_v6.rs`

Three logical sections:

### 1. Builder (`build_btree`)

```rust
pub fn build_btree<K>(
    sorted_entries: impl Iterator<Item = (K, FactRef)>,
    backend: &mut dyn StorageBackend,
    start_page_id: u64,
) -> Result<u64>  // returns root page ID
where K: Serialize + Ord
```

Algorithm:
1. Pack entries into leaf pages (fill factor 75%, linked via `next_leaf`)
2. Collect one separator key per leaf (first key of each leaf except the first)
3. Build internal node pages bottom-up from separator keys + child page IDs
4. If only one leaf, return that leaf page ID as root (no internal nodes)
5. Return root page ID

### 2. Range Scanner (`range_scan`)

```rust
pub fn range_scan<K>(
    root_page_id: u64,
    start: &K,
    end: &K,
    backend: &dyn StorageBackend,
    cache: &PageCache,
) -> Result<Vec<FactRef>>
where K: Serialize + Deserialize + Ord
```

Algorithm:
1. Traverse internal nodes from root to find the leaf containing `start`
2. Scan leaf pages following `next_leaf` chain, collecting `FactRef`s for entries where `start <= key <= end`
3. Stop when `key > end` or `next_leaf == 0`

### 3. `OnDiskIndexReader`

Implements the new `CommittedIndexReader` trait. Holds:
- `Arc<PageCache>`
- `Arc<Mutex<B>>` (backend)
- `eavt_root`, `aevt_root`, `avet_root`, `vaet_root: u64`

Each `range_scan_*` method calls the generic `range_scan` with the appropriate root page ID and key bounds.

## New Trait: `CommittedIndexReader`

Added to `src/storage/mod.rs` alongside `CommittedFactReader`:

```rust
pub trait CommittedIndexReader: Send + Sync {
    fn range_scan_eavt(&self, start: &EavtKey, end: &EavtKey) -> Result<Vec<FactRef>>;
    fn range_scan_aevt(&self, start: &AevtKey, end: &AevtKey) -> Result<Vec<FactRef>>;
    fn range_scan_avet(&self, start: &AvetKey, end: &AvetKey) -> Result<Vec<FactRef>>;
    fn range_scan_vaet(&self, start: &VaetKey, end: &VaetKey) -> Result<Vec<FactRef>>;
}
```

## `FactStorage` Changes

### Field Changes

```rust
// Before
indexes: Indexes,                                          // all facts
committed_fact_reader: Option<Arc<dyn CommittedFactReader>>,

// After
pending_indexes: Indexes,                                  // pending only
committed_fact_reader: Option<Arc<dyn CommittedFactReader>>,
committed_index_reader: Option<Arc<dyn CommittedIndexReader>>,  // new
```

### Query Merge Pattern (EAVT example)

```
start = EavtKey { entity, attribute: "", valid_from: i64::MIN, valid_to: i64::MIN, tx_count: 0 }
end   = EavtKey { entity, attribute: "\u{10FFFF}", valid_from: i64::MAX, valid_to: i64::MAX, tx_count: u64::MAX }

pending_refs   = pending_indexes.eavt.range(start..=end)  // in-memory BTreeMap
committed_refs = committed_index_reader.range_scan_eavt(start, end)  // page cache

all_refs = chain(pending_refs, committed_refs)
// resolve each FactRef → Fact:
//   page_id == 0 → pending vec
//   page_id >= 1 → CommittedFactReader (page cache)
```

Same merge pattern applies to `get_facts_by_attribute` (AEVT), value-range scans (AVET), and reverse-ref scans (VAET).

**Full scans** (no index hint): fall back to `CommittedFactReader.stream_all()` + pending vec. No index needed.

**`load_fact` (WAL replay)**: inserts into `pending_indexes` only. Unchanged.

## Checkpoint Sequence

1. Pack pending facts → new fact pages (unchanged)
2. For each of the 4 indexes:
   - If existing B+tree root ≠ 0: stream leaf pages sequentially to get sorted committed entries
   - Merge-sort with sorted pending BTreeMap entries
   - Call `build_btree(merged_iter)` → new B+tree pages, get new root page ID
3. Write `FileHeader` v6 with new root page IDs + updated `page_count`
4. Sync backend
5. Clear WAL
6. Replace `committed_index_reader` on `FactStorage` with new `OnDiskIndexReader`
7. Clear `pending_indexes`

Old index pages (paged-blob `0x11` or previous B+tree pages) are abandoned. Their page IDs are no longer referenced by the header and will never be loaded into the page cache.

---

## Auto-Checkpoint

**Trigger**: after `WriteTransaction::commit()`, if the WAL fact count exceeds `auto_checkpoint_threshold`, synchronously call `checkpoint()`.

**Configuration**: `OpenOptions::auto_checkpoint_threshold(usize)` (default: 1,000 facts; 0 = disabled).

**Implementation**: `Minigraf` stores the threshold. `commit()` checks `wal.fact_count() >= threshold` and calls `self.checkpoint()` if true. Runs on the committing thread; no background thread.

---

## v5 → v6 Migration

Migration is **eager on open** (not deferred to first checkpoint), so memory benefits take effect immediately.

**On open of a v5 file:**
1. Read old paged-blob indexes via existing `btree.rs` `read_*_index` functions → 4 in-memory BTreeMaps
2. Immediately call `build_btree` for each → write v6 B+tree pages to new page IDs
3. Write v6 header with new root page IDs
4. Set `committed_index_reader` to new `OnDiskIndexReader`
5. `pending_indexes` starts empty

**`btree.rs`** is kept read-only (no new writes) so this migration path continues to work. It can be removed in a future phase once v5 files are no longer in use.

**`FileHeader.validate()`**: updated to accept versions 1–6.

---

## `FileHeader` v6

Same 72-byte layout as v5. Only the version field changes:

```
FORMAT_VERSION = 6
```

`eavt_root_page`, `aevt_root_page`, `avet_root_page`, `vaet_root_page` continue to hold root page IDs — they pointed to paged-blob roots in v5, and now point to B+tree roots in v6.

## Testing Strategy

### Unit Tests

**`src/storage/btree_v6.rs`**:
- `build_btree` with 0, 1, and N entries — root page ID returned; leaf chain intact
- Range scan: exact match, prefix range, empty range, range spanning multiple leaf pages
- Fill-factor: pages do not exceed 3KB of data
- Multi-level tree: enough entries to force internal node creation; verify traversal
- `OnDiskIndexReader`: all four `range_scan_*` methods with `MemoryBackend`

**`src/storage/mod.rs`**:
- `FileHeader` v6 byte-layout pin test
- `validate()` accepts versions 1–6

### Integration Tests (`tests/btree_v6_test.rs`)

- **Roundtrip**: transact facts → checkpoint → close → reopen → query returns correct results
- **Range scan across leaves**: 500+ facts; EAVT range returns correct subset
- **v5→v6 migration**: create v5 file → open with v6 code → migration runs → queries work → header is v6
- **Pending + committed merge**: transact → checkpoint (committed) → transact more (pending) → query sees both
- **Auto-checkpoint**: transact facts exceeding threshold → WAL is cleared automatically
- **Dead space**: multiple checkpoints → file size grows; queries remain correct

### Regression

All 301 existing tests must continue to pass unchanged.

## Expected Impact

| Metric | Before (v5) | After (v6) |
|--------|-------------|-----------|
| Index memory at 1M facts | ~1.33 GB | O(cache_pages) ≈ 1 MB default |
| Open time | O(index_size) | O(1) |
| Checkpoint index writes | O(all_index_pages) | O(all_index_pages) — same; future incremental-insert phase will improve this |
| Startup index load | Full deserialise | Root page only |

## Out of Scope for Phase 6.5

- Incremental B+tree insert (no page splits): deferred to a future phase once bulk-rebuild is validated
- Dead-page compaction: future phase
- `lib.rs` API narrowing (pre-publish cleanup): included as a gate item but tracked separately in the pre-publishing checklist
- crates.io publish: gate on Phase 6.5 completion per ROADMAP
